# План по усилению безопасности и управляемости ServerConfiguration

## 1. Цели и границы работ

### 1.1. Цели

- Повысить предсказуемость и безопасность конфигурации серверов `server1` и `server2`.
- Снизить риски потери SSH‑доступа при работе с policy routing / nftables и full‑tunnel режимом.
- Ввести единый, формализованный порядок внесения и тестирования изменений.
- Обеспечить воспроизводимость установки/обновления через скрипты репозитория.

### 1.2. Границы

В план входят:
- только артефакты репозитория `ServerConfiguration` (скрипты, README, env‑файлы);
- сценарии установки и обновления для `server1` и `server2`;
- минимальные smoke‑тесты с использованием `curl` и утилит ОС.

В план **не входят** (могут рассматриваться отдельно):
- изменение схемы управления доступом к серверам (SSH‑ключи, VPN вне репозитория и т.п.);
- миграция инфраструктуры (смена провайдера, IP‑адресов);
- мониторинг и алертинг вне простых bash‑проверок.

## 2. Риски и меры снижения

| Риск | Описание | Меры снижения |
|------|----------|---------------|
| Потеря SSH‑доступа при full‑tunnel | Неправильный порядок применения nftables/policy routing может увести и SSH‑трафик в туннель. | Чёткий порядок действий (см. раздел 7), обязательное наличие out‑of‑band доступа, предварительный safe mode, пошаговые smoke‑тесты до переключения. |
| Нарушение существующей рабочей конфигурации | Обновления скриптов могут сломать уже настроенные сервера. | Добавление обратносуместимых изменений, сохранение старых скриптов/флагов, наличие rollback‑процедур. |
| Неполная документация | Знания остаются «в голове», невозможно воспроизвести настройки через время. | Расширенный README + этот план, ссылки между документами, чек‑листы DoD. |
| Ошибки в env‑файлах | Неверные параметры приводят к нерабочему туннелю или утечкам трафика. | Явное описание параметров `.env`, базовые валидации в скриптах (проверка переменных, типов, допустимых значений). |
| Неочевидные side‑effects iptables/nftables | Случайное затирание чужих правил. | Использование именованных таблиц/sets, аккуратное добавление правил и явная очистка только своих структур. |

## 3. Backlog по этапам (30 / 60 / 90 дней)

### 3.1. Горизонт 0–30 дней (Stabilization, High priority)

Фокус: описание текущего состояния, минимальные улучшения без изменений поведения.

1. **Документация и обзор** (H / 4–6 ч)
   - Описать текущую архитектуру связки `server1` ↔ `server2` в README и/или отдельной диаграмме.
   - Явно зафиксировать отличия safe mode vs full‑tunnel.
   - Добавить раздел «Безопасное внесение изменений» (чек‑лист) в корневой README.

2. **Формализация smoke‑тестов** (H / 3–4 ч)
   - Свести существующие команды проверки (из README) в единый раздел «Smoke tests».
   - Добавить команды проверки для каждого сценария: `server2`, `server1 safe`, `server1 full`.

3. **Базовые проверки env‑файлов** (M / 4–8 ч)
   - В `server1/setup.sh` и `server2/setup.sh` добавить проверки обязательных переменных (`set -u`, `: "${VAR:?}"`).
   - Добавить сообщения об ошибках с подсказками, какие поля проверить в `.env`.

4. **Минимальный hardening скриптов** (M / 4–6 ч)
   - Включить `set -euo pipefail` в ключевых скриптах установки (`setup.sh`, `install_*.sh`).
   - Добавить явный `trap` для логирования и отката простейших шагов, где это возможно (например, открепление временных маршрутов).

### 3.2. Горизонт 30–60 дней (Hardening & Automation)

Фокус: предсказуемость, автоматизация рутинных проверок.

1. **Расширенный hardening iptables/nftables** (H / 8–12 ч)
   - Явно документировать какие chains/sets создаёт репозиторий.
   - Добавить in‑script проверки перед модификацией правил (например, не затирать чужие таблицы).

2. **Скрипты самопроверки (healthcheck)** (M / 6–10 ч)
   - Добавить в `server1/` и `server2/` скрипты `healthcheck.sh`, которые:
     - проверяют доступность туннеля;
     - валидируют ключевые процессы (`systemctl status`/`ps`);
     - выполняют пару `curl`‑запросов через туннель.

3. **Унификация логирования** (M / 4–6 ч)
   - Ввести единообразный формат лог‑сообщений в скриптах (`[server1][safe] ...`).
   - Добавить флаг `--verbose` для детального вывода.

4. **Шаблоны `.env` с комментариями** (L / 3–4 ч)
   - Расширить `.env.example` с пояснениями к каждому параметру.
   - Добавить примеры безопасных значений (особенно для портов/сетей).

### 3.3. Горизонт 60–90 дней (Reliability & Observability)

Фокус: устойчивость, наблюдаемость и процедурность.

1. **Расширенные rollback‑сценарии** (H / 8–12 ч)
   - Выделить отдельные скрипты `rollback_*.sh` для обратимого отката правил и маршрутизации.
   - Документировать гарантированный минимум для восстановления «рабочей» конфигурации.

2. **Чек‑листы перед/после изменений (DoD)** (M / 4–6 ч)
   - Формализовать DoD для изменений в `server1`/`server2` (см. раздел 5).
   - Включить в репозиторий как часть документации для ревьюеров.

3. **Расширенные сценарии smoke‑тестов** (M / 6–8 ч)
   - Добавить тесты для типичных сервисов (HTTP/HTTPS, DNS) через туннель.
   - При необходимости — простые bash‑скрипты для пакетного прогона тестов.

4. **Рекомендации по мониторингу** (L / 4–6 ч)
   - Добавить секцию рекомендаций: какие метрики и логи стоит отслеживать (systemd‑журналы, сетевые counters).

## 4. Конкретные изменения по файлам/скриптам

> Примечание: ниже — целевой перечень изменений, который может уточняться в процессе внедрения. План ориентирован на обратную совместимость и отсутствие деструктивных правок.

### 4.1. Корень репозитория

- `README.md`
  - Добавить ссылку на данный план (`HARDENING_PLAN.md`).
  - Добавить краткий раздел «Безопасное внесение изменений» с базовыми принципами.
  - Уточнить/перенести разделы про smoke‑тесты в отдельный блок.

- `HARDENING_PLAN.md`
  - (настоящий документ) — поддерживать в актуальном состоянии по мере внедрения.

### 4.2. server1

Кандидаты на изменения:

- `server1/setup.sh`
  - Включить `set -euo pipefail`.
  - Добавить проверки обязательных переменных окружения.
  - Чётко разделить ветки `safe` / `full` с явным логированием.
  - Добавить dry‑run режим (без применения правил), если это целесообразно.

- `server1/install_safe_mode.sh`, `server1/install_full_tunnel_mode.sh`
  - Привести к единому стилю (шифты, функции, логирование).
  - Добавить idempotent‑поведение там, где возможно (повторный запуск не ломает конфигурацию).

- `server1/check_via_server2.sh`
  - Формализовать выходные коды (0 — успех, ≠0 — конкретные типы ошибок).
  - Добавить краткий отчёт по ключевым проверкам.

- Новые файлы (при необходимости):
  - `server1/healthcheck.sh` — базовая самопроверка.
  - `server1/rollback_safe.sh` / `server1/rollback_full.sh` — аккуратный откат конфигурации для соответствующего режима.

### 4.3. server2

- `server2/setup.sh`
  - Аналогично: `set -euo pipefail`, проверки env, логирование.

- Новые файлы (при необходимости):
  - `server2/healthcheck.sh` — проверка того, что Shadowsocks и связанные службы живы, nftables‑правила на месте.
  - `server2/rollback.sh` — аккуратное удаление/откат только тех правил и конфигураций, которые были созданы скриптами репозитория.

### 4.4. Скрипты верхнего уровня

- `start.sh`
  - Проверка минимальных зависимостей (bash, sudo, systemd, nftables/iptables).
  - Чёткие сообщения об ошибках и ссылками в README.

- `vpn_install/`, `mtproxy/`, `wireguard/`
  - Провести аналогичный аудит и задокументировать отдельным подпунктом (при необходимости выносом в следующий этап работ, если объём велик).

## 5. Критерии готовности (Definition of Done, DoD)

Изменение в репозитории считается **готовым к использованию на проде**, если выполняются все пункты:

1. **Документация**
   - README соответствующей директории обновлён и содержит:
     - назначение скрипта/режима;
     - пример запуска;
     - известные ограничения и риски.

2. **Проверки окружения**
   - Скрипты корректно отрабатывают отсутствие критичных утилит (nft, ip, systemctl и т.п.) и выдают понятное сообщение.

3. **Smoke‑тесты**
   - Есть чёткий набор команд для проверки (см. раздел 6), задокументированный и проверенный хотя бы один раз на тестовом окружении.

4. **Rollback**
   - Есть описанный способ возврата в рабочее состояние (автоматический скрипт или ручной чек‑лист).

5. **Безопасность SSH**
   - Для изменений, влияющих на сетевой стек, явно описано, как не потерять SSH‑доступ (см. раздел 7).

6. **Код‑стайл и предсказуемость**
   - Скрипты используют единый стиль (опции shell, формат логов, структура функций).
   - Повторный запуск скрипта не приводит к необратимому «ломанию» конфигурации без предупреждения.

## 6. Smoke‑тесты и rollback‑процедуры

### 6.1. Базовые smoke‑тесты

#### server2 (Shadowsocks)

1. Проверка сервиса:

```bash
systemctl status shadowsocks-libev --no-pager
```

2. Проверка сетевых правил (пример):

```bash
sudo nft list set inet filter ALLOWED_SPROXY
```

3. Проверка доступности с `server1` (через уже существующие скрипты/тулзы):

```bash
curl -4 --socks5-hostname <server2_ip>:<port> -s https://ifconfig.me
```

#### server1 — safe mode

1. Базовый тест туннеля:

```bash
sudo via-server2 curl -4 -s https://ifconfig.me
```

2. Тест через `check_via_server2.sh`:

```bash
sudo bash ./server1/check_via_server2.sh server1/.env safe
```

#### server1 — full‑tunnel mode

1. Проверка после переключения:

```bash
curl -4 -s https://ifconfig.me
```

2. Проверка, что SSH остаётся доступным (через отдельную сессию или out‑of‑band доступ).

### 6.2. Общие принципы rollback

> Конкретные команды будут оформлены в отдельных `rollback_*.sh` и/или разделах README. Ниже — общие принципы.

1. **Не удалять чужие правила.**
   - Rollback должен убирать только те сущности (chains, sets, routes), которые явно созданы скриптами репозитория.

2. **Минимальный набор для восстановления SSH.**
   - В документации и/или скрипте rollback должен быть явно указан минимальный набор команд, которые вернут сервер в состояние «SSH доступен, туннель отключён».

3. **Идемпотентность.**
   - Повторный запуск rollback не должен ломать конфигурацию.

4. **Журналирование.**
   - Rollback‑скрипты должны логировать список шагов, которые они выполняют.

## 7. Пошаговый порядок внедрения без потери SSH‑доступа

Ниже — общий порядок для безопасного перехода от текущего состояния к обновлённым скриптам и/или включению full‑tunnel.

### 7.1. Общие предпосылки

- Есть **альтернативный канал доступа** (консоль провайдера, отдельный VPN, второй SSH‑хост).
- Есть сохранённая «рабочая» конфигурация (резервные копии `.env`, экспорт текущих nftables/iptables и маршрутов).

### 7.2. Рекомендуемый порядок действий

1. **Подготовка**
   - Обновить локальный клон репозитория `ServerConfiguration`.
   - Перейти на согласованную ветку/релиз.
   - Проверить, что `.env` файлы актуальны, сделать их копии (`*.backup`).

2. **Обновление `server2`**
   - На `server2` применить обновлённый `setup.sh` **без изменения внешних портов**.
   - Выполнить smoke‑тесты `server2` (раздел 6.1).
   - При неуспехе — использовать rollback для отката только правил/настроек туннеля.

3. **Обновление `server1` в safe mode**
   - На `server1` запустить `start.sh` и убедиться, что базовые зависимости в порядке.
   - Выполнить установку/обновление safe mode через `server1/setup.sh safe server1/.env`.
   - Прогнать smoke‑тесты safe mode и убедиться, что трафик через туннель ходит, при этом SSH остаётся стабильным.

4. **Наблюдение**
   - Оставить сервер в safe mode на разумный период (от часов до суток) для наблюдения.
   - При необходимости — скорректировать параметры `.env` и повторить установку.

5. **Переход в full‑tunnel (опционально)**
   - Убедиться в наличии out‑of‑band доступа.
   - Выполнить `server1/setup.sh full server1/.env`.
   - Сразу же выполнить smoke‑тесты full‑tunnel и в отдельной сессии проверить SSH.
   - При любых аномалиях — использовать rollback‑сценарий, возвращающий сервер к safe mode или к «чистому» состоянию без туннеля.

6. **Финализация**
   - Зафиксировать финальные значения `.env` и экспортировать рабочие nftables/маршруты.
   - Обновить документацию, если по ходу были найдены отклонения от плана.

## 9. Итоги

Данный план описывает поэтапное усиление безопасности и управляемости конфигураций в репозитории `ServerConfiguration` без ломки существующей рабочей схемы. Основные принципы:

- не ломать уже рабочие сценарии;
- двигаться от документации и минимальных проверок к полноценным rollback‑и healthcheck‑механизмам;
- отдельно контролировать сценарии, потенциально влияющие на SSH‑доступ (особенно full‑tunnel);
- поддерживать документацию в актуальном состоянии и использовать её как чек‑лист при каждом изменении.
