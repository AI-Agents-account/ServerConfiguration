# План по усилению безопасности и управляемости ServerConfiguration

## 1. Цели и границы работ

### 1.1. Цели

- Повысить предсказуемость и безопасность конфигурации серверов `server1` и `server2`.
- Снизить риски потери SSH‑доступа при работе с policy routing / nftables и full‑tunnel режимом.
- Ввести единый, формализованный порядок внесения и тестирования изменений.
- Обеспечить воспроизводимость установки/обновления через скрипты репозитория.

### 1.2. Границы

В план входят:
- только артефакты репозитория `ServerConfiguration` (скрипты, README, env‑файлы);
- сценарии установки и обновления для `server1` и `server2`;
- минимальные smoke‑тесты с использованием `curl` и утилит ОС.

В план **не входят** (могут рассматриваться отдельно):
- изменение схемы управления доступом к серверам (SSH‑ключи, VPN вне репозитория и т.п.);
- миграция инфраструктуры (смена провайдера, IP‑адресов);
- мониторинг и алертинг вне простых bash‑проверок.

## 2. Риски и меры снижения

| Риск | Описание | Меры снижения |
|------|----------|---------------|
| Потеря SSH‑доступа при full‑tunnel | Неправильный порядок применения nftables/policy routing может увести и SSH‑трафик в туннель. | Чёткий порядок действий (см. раздел 7), обязательное наличие out‑of‑band доступа, предварительный safe mode, пошаговые smoke‑тесты до переключения. |
| Нарушение существующей рабочей конфигурации | Обновления скриптов могут сломать уже настроенные сервера. | Добавление обратносуместимых изменений, сохранение старых скриптов/флагов, наличие rollback‑процедур. |
| Неполная документация | Знания остаются «в голове», невозможно воспроизвести настройки через время. | Расширенный README + этот план, ссылки между документами, чек‑листы DoD. |
| Ошибки в env‑файлах | Неверные параметры приводят к нерабочему туннелю или утечкам трафика. | Явное описание параметров `.env`, базовые валидации в скриптах (проверка переменных, типов, допустимых значений). |
| Неочевидные side‑effects iptables/nftables | Случайное затирание чужих правил. | Использование именованных таблиц/sets, аккуратное добавление правил и явная очистка только своих структур. |

## 3. Backlog по этапам (30 / 60 / 90 дней)

### 3.1. Горизонт 0–30 дней (Stabilization, High priority)

Фокус: описание текущего состояния, минимальные улучшения без изменений поведения.

1. **Документация и обзор** (H / 4–6 ч)
   - Описать текущую архитектуру связки `server1` ↔ `server2` в README и/или отдельной диаграмме.
   - Явно зафиксировать отличия safe mode vs full‑tunnel.
   - Добавить раздел «Безопасное внесение изменений» (чек‑лист) в корневой README.

2. **Формализация smoke‑тестов** (H / 3–4 ч)
   - Свести существующие команды проверки (из README) в единый раздел «Smoke tests».
   - Добавить команды проверки для каждого сценария: `server2`, `server1 safe`, `server1 full`.

3. **Базовые проверки env‑файлов** (M / 4–8 ч)
   - В `server1/setup.sh` и `server2/setup.sh` добавить проверки обязательных переменных (`set -u`, `: "${VAR:?}"`).
   - Добавить сообщения об ошибках с подсказками, какие поля проверить в `.env`.

4. **Минимальный hardening скриптов** (M / 4–6 ч)
   - Включить `set -euo pipefail` в ключевых скриптах установки (`setup.sh`, `install_*.sh`).
   - Добавить явный `trap` для логирования и отката простейших шагов, где это возможно (например, открепление временных маршрутов).

### 3.2. Горизонт 30–60 дней (Hardening & Automation)

Фокус: предсказуемость, автоматизация рутинных проверок.

1. **Расширенный hardening iptables/nftables** (H / 8–12 ч)
   - Явно документировать какие chains/sets создаёт репозиторий.
   - Добавить in‑script проверки перед модификацией правил (например, не затирать чужие таблицы).

2. **Скрипты самопроверки (healthcheck)** (M / 6–10 ч)
   - Добавить в `server1/` и `server2/` скрипты `healthcheck.sh`, которые:
     - проверяют доступность туннеля;
     - валидируют ключевые процессы (`systemctl status`/`ps`);
     - выполняют пару `curl`‑запросов через туннель.

3. **Унификация логирования** (M / 4–6 ч)
   - Ввести единообразный формат лог‑сообщений в скриптах (`[server1][safe] ...`).
   - Добавить флаг `--verbose` для детального вывода.

4. **Шаблоны `.env` с комментариями** (L / 3–4 ч)
   - Расширить `.env.example` с пояснениями к каждому параметру.
   - Добавить примеры безопасных значений (особенно для портов/сетей).

### 3.3. Горизонт 60–90 дней (Reliability & Observability)

Фокус: устойчивость, наблюдаемость и процедурность.

1. **Расширенные rollback‑сценарии** (H / 8–12 ч)
   - Выделить отдельные скрипты `rollback_*.sh` для обратимого отката правил и маршрутизации.
   - Документировать гарантированный минимум для восстановления «рабочей» конфигурации.

2. **Чек‑листы перед/после изменений (DoD)** (M / 4–6 ч)
   - Формализовать DoD для изменений в `server1`/`server2` (см. раздел 5).
   - Включить в репозиторий как часть документации для ревьюеров.

3. **Расширенные сценарии smoke‑тестов** (M / 6–8 ч)
   - Добавить тесты для типичных сервисов (HTTP/HTTPS, DNS) через туннель.
   - При необходимости — простые bash‑скрипты для пакетного прогона тестов.

4. **Рекомендации по мониторингу** (L / 4–6 ч)
   - Добавить секцию рекомендаций: какие метрики и логи стоит отслеживать (systemd‑журналы, сетевые counters).

## 5. Критерии готовности (Definition of Done, DoD)

Изменение в репозитории считается **готовым к использованию на проде**, если выполняются все пункты:

1. **Документация**
   - README соответствующей директории обновлён и содержит:
     - назначение скрипта/режима;
     - пример запуска;
     - известные ограничения и риски.

2. **Проверки окружения**
   - Скрипты корректно отрабатывают отсутствие критичных утилит (nft, ip, systemctl и т.п.) и выдают понятное сообщение.

3. **Smoke‑тесты**
   - Есть чёткий набор команд для проверки (см. раздел 6), задокументированный и проверенный хотя бы один раз на тестовом окружении.

4. **Rollback**
   - Есть описанный способ возврата в рабочее состояние (автоматический скрипт или ручной чек‑лист).

5. **Безопасность SSH**
   - Для изменений, влияющих на сетевой стек, явно описано, как не потерять SSH‑доступ (см. раздел 7).

6. **Код‑стайл и предсказуемость**
   - Скрипты используют единый стиль (опции shell, формат логов, структура функций).
   - Повторный запуск скрипта не приводит к необратимому «ломанию» конфигурации без предупреждения.

## 6. Smoke‑тесты и rollback‑процедуры

### 6.1. Базовые smoke‑тесты

#### server2 (Shadowsocks)

1. Проверка сервиса:

```bash
systemctl status shadowsocks-libev --no-pager
```

2. Проверка сетевых правил (пример):

```bash
sudo nft list set inet filter ALLOWED_SPROXY
```

3. Проверка доступности с `server1` (через уже существующие скрипты/тулзы):

```bash
curl -4 --socks5-hostname <server2_ip>:<port> -s https://ifconfig.me
```

#### server1 — safe mode

1. Базовый тест туннеля:

```bash
sudo via-server2 curl -4 -s https://ifconfig.me
```

2. Тест через `check_via_server2.sh`:

```bash
sudo bash ./server1/check_via_server2.sh server1/.env safe
```

#### server1 — full‑tunnel mode

1. Проверка после переключения:

```bash
curl -4 -s https://ifconfig.me
```

2. Проверка, что SSH остаётся доступным (через отдельную сессию или out‑of‑band доступ).

### 6.2. Общие принципы rollback

> Конкретные команды будут оформлены в отдельных `rollback_*.sh` и/или разделах README. Ниже — общие принципы.

1. **Не удалять чужие правила.**
   - Rollback должен убирать только те сущности (chains, sets, routes), которые явно созданы скриптами репозитория.

2. **Минимальный набор для восстановления SSH.**
   - В документации и/или скрипте rollback должен быть явно указан минимальный набор команд, которые вернут сервер в состояние «SSH доступен, туннель отключён».

3. **Идемпотентность.**
   - Повторный запуск rollback не должен ломать конфигурацию.

4. **Журналирование.**
   - Rollback‑скрипты должны логировать список шагов, которые они выполняют.

## 7. Пошаговый порядок внедрения без потери SSH‑доступа

Ниже — общий порядок для безопасного перехода от текущего состояния к обновлённым скриптам и/или включению full‑tunnel.

### 7.1. Общие предпосылки

- Есть **альтернативный канал доступа** (консоль провайдера, отдельный VPN, второй SSH‑хост).
- Есть сохранённая «рабочая» конфигурация (резервные копии `.env`, экспорт текущих nftables/iptables и маршрутов).

### 7.2. Рекомендуемый порядок действий

1. **Подготовка**
   - Обновить локальный клон репозитория `ServerConfiguration`.
   - Перейти на согласованную ветку/релиз.
   - Проверить, что `.env` файлы актуальны, сделать их копии (`*.backup`).

2. **Обновление `server2`**
   - На `server2` применить обновлённый `setup.sh` **без изменения внешних портов**.
   - Выполнить smoke‑тесты `server2` (раздел 6.1).
   - При неуспехе — использовать rollback для отката только правил/настроек туннеля.

3. **Обновление `server1` в safe mode**
   - На `server1` запустить `start.sh` и убедиться, что базовые зависимости в порядке.
   - Выполнить установку/обновление safe mode через `server1/setup.sh safe server1/.env`.
   - Прогнать smoke‑тесты safe mode и убедиться, что трафик через туннель ходит, при этом SSH остаётся стабильным.

4. **Наблюдение**
   - Оставить сервер в safe mode на разумный период (от часов до суток) для наблюдения.
   - При необходимости — скорректировать параметры `.env` и повторить установку.

5. **Переход в full‑tunnel (опционально)**
   - Убедиться в наличии out‑of‑band доступа.
   - Выполнить `server1/setup.sh full server1/.env`.
   - Сразу же выполнить smoke‑тесты full‑tunnel и в отдельной сессии проверить SSH.
   - При любых аномалиях — использовать rollback‑сценарий, возвращающий сервер к safe mode или к «чистому» состоянию без туннеля.

6. **Финализация**
   - Зафиксировать финальные значения `.env` и экспортировать рабочие nftables/маршруты.
   - Обновить документацию, если по ходу были найдены отклонения от плана.

## 9. Разделение трафика на server1 по региону/направлению сервиса

### 9.1. Цели сценария

Цель: на `server1` реализовать **селективный вывод трафика**:

- трафик к российским ресурсам и внутренним серверам идёт напрямую через основной WAN‑шлюз;
- трафик к зарубежным ресурсам уходит в туннель (`tun0` → `server2` → Shadowsocks);
- SSH‑доступ к `server1` **никогда** не попадает под случайный редирект в туннель.

Режим разделения трафика рассматривается как отдельный, дополняющий уже описанные `safe` и `full` режимы:

- может быть реализован как **отдельный профиль** (`mode=split`),
- либо как надстройка над `full` с дополнительными правилами исключений для «российских»/внутренних направлений.

### 9.2. Целевая архитектура на высоком уровне

Ключевые элементы:

- **Интерфейсы**:
  - `eth0` (или аналогичный) — основной WAN‑интерфейс `server1`;
  - `tun0` — интерфейс туннеля (`tun2socks`), ведущий к `server2`.
- **Таблицы маршрутизации**:
  - `main` — стандартная таблица маршрутов, используемая по умолчанию (WAN);
  - `tun` — отдельная таблица, в которой default‑маршрут идёт через `tun0`;
  - при необходимости — дополнительная таблица `wan` с явным default‑маршрутом через WAN.
- **Маркировка трафика**:
  - пакеты, предназначенные для зарубежных ресурсов, маркируются (`fwmark`) и отправляются в таблицу `tun` с помощью `ip rule`;
  - трафик к российским/внутренним ресурсам не маркируется и продолжает использовать таблицу `main`/`wan`.
- **DNS‑разрешение**:
  - трафик классифицируется либо по IP‑сети (ipset с префиксами), либо по доменам (dnsmasq + domain/ipset);
  - важно обеспечить предсказуемость: какой DNS используется и какие правила сопоставления домен → IP → таблица маршрутизации.

Дополнительные требования:

- явные **исключения для SSH** (порт, подсети управляющих хостов);
- чётко задокументированный список таблиц/sets/цепочек, создаваемых репозиторием;
- единообразные имена (`SC_SPLIT_FOREIGN`, `SC_SPLIT_RU`, `SC_SPLIT_MARK_TUN` и т.п.).

### 9.3. Варианты реализации

#### 9.3.1. Вариант A: ipset + nftables + policy routing (по IP‑сетям)

Основная идея:

1. **Сформировать ipset/сет для «российских» сетей** (и/или наоборот — «зарубежных»).
2. В `nftables` создать отдельную таблицу/цепочку, которая:
   - для пакетов к адресам из `RU_SET` **не** ставит метку (используется WAN);
   - для остальных пакетов выставляет `meta mark set <mark_tun>`.
3. Через `ip rule` перенаправлять трафик с меткой `<mark_tun>` в таблицу маршрутизации `tun`.

Плюсы:

- гибкий контроль по IP‑сетям, можно импортировать готовые списки (например, `*.txt` с префиксами);
- хорошо стыкуется с уже планируемым hardening `nftables`/policy routing (см. разделы 3, 4, 7); 
- минимизирует зависимости от конкретного DNS.

Минусы/ограничения:

- требуется поддерживать актуальность списка префиксов (динамика IP, появление новых CDN‑узлов);
- географическая принадлежность IP не всегда соответствует «стране ресурса» (см. риски 9.5).

Рекомендуемые шаги реализации:

- в `server1/` добавить скрипт (или подпункты в `setup.sh`), который:
  - создаёт именованные sets (например, `SC_RU_NETS`, `SC_TUN_EXCLUDED_NETS`);
  - наполняет их из файлов `server1/ipsets/ru.conf`, `server1/ipsets/local.conf` и т.п.;
  - настраивает `nft`‑цепочки вида `sc_split_prerouting`, `sc_split_output`;
  - устанавливает `ip rule add fwmark <TUN_MARK> lookup tun`.

#### 9.3.2. Вариант B: dnsmasq + domain‑sets + policy routing (по доменным именам)

Основная идея:

1. Развернуть локальный резолвер (например, `dnsmasq`) на `server1`.
2. Определить **domain‑sets** для российских доменов и критичных внутренних ресурсов:
   - списки вида `ru-domains.list`, `bank-internal.list`, `domestic-critical.list`;
3. Настроить `dnsmasq` так, чтобы при резолве доменов из этих списков соответствующие IP попадали в ipset (например, `SC_RU_DOMAINS_SET`).
4. Дальше — тот же механизм, что в варианте A: `nftables` + `ip rule` на основе ipset.

Плюсы:

- классификация на уровне доменов (ближе к прикладной логике, проще обсуждать с бизнесом/безопасностью);
- можно точечно прописывать правила для конкретных сервисов (например, отдельный CDN‑домен).

Минусы/ограничения:

- риски, связанные с DNS (poisoning, split‑horizon, кэширование);
- сложнее отлаживать (IP‑адреса могут меняться, для отладки нужны логи DNS/sets).

Рекомендуемые элементы реализации:

- конфигурационный файл `server1/dnsmasq.conf` с:
  - включёнными логами для отладки (`log-queries`, `log-facility`);
  - `ipset=/example.ru/SC_RU_DOMAINS_SET` style‑правилами;
- шаблоны списков доменов (`server1/domains/ru.txt`, `server1/domains/foreign_force_tun.txt`);
- интеграция с существующими smoke‑тестами (см. 9.6).

#### 9.3.3. Вариант C: policy routing по пользователям/портам (комбинированный подход)

Дополнительный/резервный вариант, который можно комбинировать с A и B:

- выделить системного пользователя/группу (аналог `tunroute` в safe mode);
- весь трафик этого пользователя по умолчанию выводить через `tun0`;
- для отдельных служб/портов (например, локальный кэш, monitoring‑агент) оставить WAN.

Подходит для сценариев, где критично разделение не столько по стране, сколько по типу сервиса (например, браузер/CLI через туннель, системные обновления — напрямую).

### 9.4. Порядок внедрения split‑routing без потери SSH

Рекомендуемый общий порядок (дополнение к разделу 7):

1. **Защита SSH до любых изменений маршрутизации**
   - Явно зафиксировать, какой IP/интерфейс/порт используется для SSH‑доступа.
   - В `nftables` добавить правило в самом верху подходящей цепочки: 
     - не изменять и не перенаправлять трафик к порту SSH (например, `tcp dport 22`),
     - и/или к IP управляющих хостов/VPN.
   - При необходимости — забиндить `sshd` на конкретный адрес WAN‑интерфейса.

2. **Подготовка таблиц маршрутизации и `ip rule`**
   - Создать таблицу `tun` и, при необходимости, `wan` в `/etc/iproute2/rt_tables`.
   - Добавить в `tun` default‑маршрут через `tun0`.
   - Добавить в `wan` явный default‑маршрут через WAN.
   - Проверить, что при отсутствии меток трафик ведёт себя как раньше (WAN по умолчанию).

3. **Внедрение ipset/sets без включения редиректа**
   - Создать пустые sets (`SC_RU_NETS`, `SC_RU_DOMAINS_SET` и т.п.).
   - Подключить `dnsmasq` (если используется) и убедиться, что он не ломает текущий DNS (временно без ipset‑правил).
   - Добавить `nft`‑цепочки в режиме **log‑only** (логируют, какие пакеты *были бы* помечены, но не выставляют mark).

4. **Постепенное включение маркировки трафика**
   - Включить выставление `fwmark` только для ограниченного подмножества (например, CURL‑тестов локально или трафика от отдельного пользователя).
   - Проверить, что:
     - SSH и управляющие сессии не затрагиваются;
     - CURL к российским ресурсам идёт через WAN;
     - CURL к зарубежным ресурсам идёт через `tun0`.

5. **Расширение объёма трафика**
   - Постепенно расширять списки ipset/domain‑sets (добавлять больше сервисов), контролируя логи.
   - При каждом расширении — прогонять smoke‑тесты (см. 9.6).

6. **Фиксация и документирование финальной конфигурации**
   - Обновить `.env`‑параметры, описать режим `split` в `server1/README.md`.
   - Экспортировать рабочие правила `nft`/`ip rule` как эталон (для rollback и воспроизведения).

### 9.5. Основные риски и особенности

1. **Geo‑CDN и «плавающие» IP‑адреса**
   - Один и тот же домен может резолвиться в разные IP (в т.ч. разные страны) в зависимости от региона, оператора, точки входа.
   - Запись домена в «российский» список не гарантирует, что все его IP останутся локальными.

2. **Динамические IP и внешние списки**
   - Публичные списки префиксов могут устаревать;
   - собственные списки требуют регулярного обновления и пересмотра.

3. **DNS poisoning / подмена ответов**
   - При использовании domain‑sets злоумышленник, контролирующий DNS, может:
     - подмешать «неожиданный» IP в ipset;
     - направить чувствительный трафик в обход ожидаемого маршрута.
   - Меры снижения:
     - использовать надёжные резолверы (DoT/DoH, DNSSEC при наличии);
     - ограничивать доверие к публичным DNS, особенно внутри туннеля;
     - включить логирование DNS‑запросов и периодически их анализировать.

4. **Split‑horizon DNS**
   - Один и тот же домен может резолвиться по‑разному внутри и снаружи туннеля.
   - Это усложняет отладку (разные IP → разные маршруты).

5. **Отказ туннеля**
   - При падении `tun0` трафик с меткой `TUN_MARK` может:
     - либо «вывалиться» в black‑hole (если так настроено);
     - либо неожиданно пойти через WAN (если fallback неаккуратно прописан).
   - Важно явно определить желаемое поведение (обычно: **fail‑closed** для зарубежного трафика, но не ломать критичные российские/внутренние сервисы).

6. **Человеческий фактор**
   - Сложные связки ipset + nftables + ip rule труднее понимать и сопровождать.
   - Нужна хорошая документация и простые команды диагностики.

### 9.6. Smoke‑тесты для режима разделения трафика

Примеры базовых тестов (дополнение к разделу 6.1):

1. **Проверка базовой связности и SSH**

```bash
# Выполняется с управляющего хоста
ssh user@server1 'echo OK && ip route get 1.1.1.1'
```

Ожидаемый результат:

- SSH‑подключение стабильно;
- маршрут до 1.1.1.1 (зарубежный IP) показывает использование таблицы `tun`/интерфейса `tun0` *после* включения split‑режима.

2. **Тест российского ресурса через WAN**

```bash
ssh user@server1 'curl -4 -s https://ya.ru -o /dev/null -w "%{remote_ip} %{http_code}\\n"'
```

Ожидаемый результат:

- соединение устанавливается;
- по логам `nft`/ipset видно, что пакет не попал под `TUN_MARK`;
- трассировка (`mtr`/`traceroute`) подтверждает маршрут через WAN.

3. **Тест зарубежного ресурса через туннель**

```bash
ssh user@server1 'curl -4 -s https://ifconfig.me -o /dev/null -w "%{remote_ip} %{http_code}\\n"'
```

Ожидаемый результат:

- внешний IP соответствует `server2`/провайдеру туннеля;
- в `ip route get` для IP `ifconfig.me` видно использование таблицы `tun`/интерфейса `tun0`.

4. **Диагностика ipset/nftables**

```bash
ssh user@server1 '
  sudo nft list ruleset | sed -n "/sc_split/,+20p" && \
  sudo nft list set inet sc_split SC_RU_NETS && \
  sudo nft list set inet sc_split SC_FOREIGN_NETS
'
```

Цель:

- убедиться, что sets существуют, наполняются и используются в правилах;
- проверить, что нет конфликтующих правил, перехватывающих SSH.

5. **Проверка отказоустойчивости**

```bash
# Симулировать падение туннеля
ssh user@server1 'sudo systemctl stop tun2socks || true'

# Попробовать обратиться к зарубежному ресурсу
ssh user@server1 'curl -4 -s https://ifconfig.me --max-time 10 || echo "failed as expected"'
```

Ожидаемый результат:

- поведение соответствует выбранной политике (fail‑closed или явный fallback);
- SSH‑доступ к `server1` остаётся рабочим.

### 9.7. Rollback для режима разделения трафика

Отдельно от общих rollback‑принципов (раздел 6.2) рекомендуется:

1. **Выделить отдельный rollback‑скрипт**, например `server1/rollback_split.sh`, который:
   - удаляет/отключает только цепочки `sc_split_*` и связанные `ip rule`;
   - не трогает остальные таблицы/цепочки, созданные для safe/full режимов;
   - по возможности, возвращает трафик к исходному поведению «всё через WAN».

2. **Поддерживать режим "disable" без удаления конфигурации**

- Скрипт должен уметь:
  - выключать применение правил (например, отключать jump из основной цепочки в `sc_split_*`);
  - оставлять сами sets и вспомогательные структуры для последующего включения.

3. **Минимальный чек‑лист ручного отката** (на случай, если скрипты недоступны):

- удалить/закомментировать `ip rule`, ссылающиеся на `fwmark=TUN_MARK` или таблицу `tun`;
- отключить/удалить `nft`‑цепочки с маркировкой split‑режима;
- убедиться, что default‑маршрут в таблице `main`/`wan` указывает на WAN;
- перезапустить `sshd` (если менялись bind‑адреса) и проверить доступ.

4. **Smoke‑тест rollback**

- после выполнения rollback‑скрипта прогнать базовые smoke‑тесты из раздела 6.1 для safe/WAN‑режима;
- убедиться, что:
  - российские и зарубежные ресурсы доступны хотя бы через WAN;
  - SSH‑доступ стабилен;
  - дополнительные правила `sc_split_*` действительно не применяются.

## 10. Итоги

Данный план описывает поэтапное усиление безопасности и управляемости конфигураций в репозитории `ServerConfiguration` без ломки существующей рабочей схемы. Основные принципы:

- не ломать уже рабочие сценарии;
- двигаться от документации и минимальных проверок к полноценным rollback‑и healthcheck‑механизмам;
- отдельно контролировать сценарии, потенциально влияющие на SSH‑доступ (особенно full‑tunnel);
- поддерживать документацию в актуальном состоянии и использовать её как чек‑лист при каждом изменении.
